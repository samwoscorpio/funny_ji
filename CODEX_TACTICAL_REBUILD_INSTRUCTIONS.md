# 5. 地图系统

## 5.1 地图尺寸

使用固定 `7 × 7` 方格地图。

坐标：

- 列：1～7
- 行：A～G
- 内部数据统一使用 `{ row: 0..6, col: 0..6 }`

## 5.2 初始地图

定义一张硬编码测试地图，不做随机生成。

建议数据：

```js
const TACTICAL_MAP = {
  rows: 7,
  cols: 7,
  playerSpawn: { row: 5, col: 1 },
  enemySpawn: { row: 1, col: 5 },
  objective: { row: 3, col: 3 },
  energyTiles: [
    { row: 1, col: 1 },
    { row: 5, col: 5 }
  ],
  walls: [
    { row: 2, col: 1 },
    { row: 2, col: 2 },
    { row: 2, col: 4 },
    { row: 2, col: 5 },
    { row: 4, col: 1 },
    { row: 4, col: 2 },
    { row: 4, col: 4 },
    { row: 4, col: 5 }
  ],
  bushes: [
    { row: 0, col: 5 },
    { row: 5, col: 2 }
  ]
};
```

允许根据 UI 实际效果轻微调整，但必须保持：

- 双方出生点；
- 一个中央据点；
- 两个能量点；
- 若干墙；
- 至少两个草丛；
- 地图左右大体平衡。

## 5.3 地形

第一版只实现：

### floor

普通可通行地面。

### wall

不可进入，不可穿过。

### bush

可进入。第一版仅显示地形，不实现隐身。

### energy

可进入。回合结束时，站在能量格上的单位获得 `+1 XP`。

### objective

可进入。用于据点计分。

# 7. 移动系统

## 7.1 基础移动力

每名英雄每回合默认 `3` 点移动力。

本阶段不根据英雄职业改变移动力。

允许：

- 上；
- 下；
- 左；
- 右。

不允许斜向移动。

## 7.2 玩家选择路径

交互：

1. 点击玩家单位，或默认玩家单位已选中；
2. 地图显示当前可到达格；
3. 点击相邻合法格，添加一步路径；
4. 继续点击相邻合法格，延长路径；
5. 点击路径中的较早格，截断路径；
6. 点击玩家当前格，清空路径；
7. 不允许越过墙；
8. 不允许超过移动力；
9. 不允许路径经过敌方当前所在格；
10. 暂时不处理“经过敌方计划位置”。

在地图上显示：

- 可移动范围；
- 已选路径；
- 路径序号或箭头；
- 终点高亮。

## 7.3 AI 移动计划

先实现简单可靠 AI：

优先级：

1. 如果能在本回合到达可攻击距离，则向玩家靠近；
2. 否则如果中央据点未被自己稳定占领，则向中央据点靠近；
3. 否则向最近能量点靠近；
4. 如果无合法路径，则原地不动。

只需 BFS 求 7×7 最短路径。

AI 计划必须在玩家锁定行动后生成，避免 AI 读取玩家已经选择的最终目标并精准作弊。

AI 可以依据玩家“当前公开位置”决策，但不能依据玩家本回合计划路径做针对性选择。

# 10. 同时行动与结算流程

## 10.1 计划阶段

玩家依次完成：

1. 选择移动路径；
2. 选择行动卡；
3. 如需要则选择目标格；
4. 查看行动预览；
5. 点击“锁定行动”。

锁定前允许任意修改。

## 10.2 锁定后

1. 禁用玩家地图和手牌交互；
2. AI 生成移动与动作；
3. 进入“结算阶段”；
4. 同时展示双方计划；
5. 执行移动；
6. 结算动作；
7. 处理地形；
8. 更新据点；
9. 判断胜负；
10. 进入下一回合。

## 10.3 同时移动冲突

采用简单确定性规则：

### 双方终点不同

各自移动到终点。

### 双方终点相同

双方都停在路径的上一个合法格。

如果上一个合法格也冲突，则继续回退，直到找到不冲突位置或回到起点。

### 双方交换位置

如果玩家终点是敌人起点，同时敌人终点是玩家起点：

- 允许交换位置。

### 路径交叉

只要终点不相同，第一版允许路径交叉，不做中途碰撞。

将这一规则封装为：

```js
resolveSimultaneousMovement(playerPlan, enemyPlan, state)
```

不要把冲突判断散落在 UI 代码中。

## 10.4 动作顺序

移动完成后，再使用双方旧版动作进行同时结算。

原则：

- 尽量复用当前现有一回合结算函数；
- 将“旧版双方 action”作为输入；
- 在调用旧结算前先判断地图攻击是否命中；
- 攻击落空时仍支付费用，但将该攻击的有效 power 视为 0；
- 防御和 Ji 不受距离影响；
- 双方攻击可同时命中；
- 不要改成传统先手制。

如果当前旧结算函数严重耦合 DOM，请先把纯规则抽成函数，再由 UI 调用。

推荐结构：

```js
function buildResolvedAction(plan, self, opponent) {}
function resolveTacticalRound(state, playerPlan, enemyPlan) {}
function applyLegacyCombatResolution(player, enemy, playerAction, enemyAction) {}
```

# 11. 据点与能量点

## 11.1 能量点

回合结束时：

- 玩家站在能量点：玩家 `+1 XP`
- 敌人站在能量点：敌人 `+1 XP`

不得超过项目原有 XP 上限逻辑；如果旧代码本身允许超过，则保持旧行为，不另行改平衡。

日志示例：

- `刺客占据能量点，获得 1 XP。`

## 11.2 中央据点

采用简化计分：

- 回合结束时只有一方站在中央据点，该方获得 `1` 点据点分；
- 双方都不在，不得分；
- 双方不可能位于同一格；
- 先得到 `4` 分的一方获胜；
- HP 归零仍按原规则立即失败。

顶部显示：

```text
据点分 2 : 1
```

击杀胜利和据点胜利都要写入战斗日志。

历史记录中：

- 仍使用现有 `win` / `loss` / `draw`；
- 可以新增 `winReason: "knockout" | "objective"`；
- 若不想改历史结构，也至少在日志里说明胜利原因。

# 13. 地图交互状态机

请显式维护交互模式，不要依赖大量互相冲突的布尔变量。

例如：

```js
const interaction = {
  mode: "planning", // planning | resolving | finished
  selection: "move", // move | card | target
  selectedCardId: null,
  selectedTarget: null,
  path: []
};
```

建议流程：

- 初始：`selection = "move"`
- 选择卡牌后：
  - 不需目标：保持或回到 move；
  - 需要目标：`selection = "target"`
- 点击“取消计划”：清空 path、card、target
- 结算中：`mode = "resolving"`
- 对局结束：`mode = "finished"`

地图格点击必须根据当前 selection 决定行为。
